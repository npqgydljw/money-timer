# PRD 生成助手

## 任务目标
- 本 Skill 用于：将产品构思或需求描述转化为结构完整、细节充分、可直接执行的 PRD 文档
- 能力包含：功能模块拆解、交互细节设计、字段规范定义、业务逻辑梳理、权限角色明确
- 触发条件：用户提出"帮我写PRD"、"生成产品需求文档"、"整理需求"等诉求，或上传需求说明文档

## 前置准备
- 无需额外依赖

## 操作步骤
- 标准流程：
  1. 收集产品信息
     - 如果用户提供了需求文档（Word/PDF/TXT等），先读取并提取核心需求
     - 如果是口头描述，引导用户补充关键信息：产品目标、核心功能、目标用户、使用场景
     - 整理产品基本信息：产品名称、定位、目标用户、核心价值

  2. 功能模块拆解
     - 根据需求拆分出独立的功能模块（如用户管理、订单管理、统计分析）
     - 识别公共模块（如权限管理、字典管理、日志记录）
     - 确定模块之间的依赖关系

  3. 逐模块生成 PRD 内容（遵循标准结构）
     - 参考 [references/prd-template.md](references/prd-template.md) 了解完整的 5 部分结构
     - 对每个功能模块，依次生成：
       * 需求描述：功能目的、业务场景、价值说明
       * 前置条件：使用该功能需要满足的先决条件
       * 交互说明：核心部分，详细描述 UI 和操作流程（参考下文细化要求）
       * 后置条件：功能执行后的状态变化
       * 异常说明：错误处理和异常情况

  4. 交互说明细化（关键步骤）
     - 查询条件：列出所有筛选字段（名称、类型、默认值、是否可清空、单选/多选）
     - 列表字段：明确展示字段、排序规则、分页方式
     - 操作权限：详细说明每个按钮的交互逻辑（弹窗、跳转、确认提示）
     - 表单字段：每个字段必须包含（参考 [references/field-spec.md](references/field-spec.md)）：
       * 字段名称
       * 字段类型（输入框、下拉、日期、级联等）
       * 必填性（必填/选填）
       * 字段限制（长度、格式、取值范围）
       * 默认值
     - 业务逻辑：对复杂流程使用条件分支描述（如登录的多层判断）

  5. 审核和完善
     - 检查每个模块是否包含完整的 5 个部分
     - 验证交互说明是否细化到字段级别
     - 确认权限和角色是否明确
     - 参考 [references/examples.md](references/examples.md) 中的完整示例进行对比

- 可选分支：
  - 当用户已提供详细需求文档：优先提取和结构化，减少询问
  - 当需求不完整时：先生成核心功能的 PRD，标记待补充部分

## 注意事项
- 仅在需要时读取参考文档，保持上下文简洁
- 交互说明必须细化到字段级别，避免模糊描述（如"用户输入信息"应细化为"用户输入手机号，11位数字，必填"）
- 复杂业务逻辑使用条件分支清晰描述（如"如果...则..."）
- 明确区分不同角色的操作权限
- 充分利用智能体的语言理解和推理能力，将用户的需求转化为规范的 PRD 格式

## 使用示例

### 示例 1：从产品描述生成 PRD
- **功能说明**：用户口头描述产品需求，智能体生成完整 PRD
- **执行方式**：智能体主导（使用自然语言指导）
- **关键指导要点**：
  1. 先引导用户补充产品核心信息
  2. 拆解功能模块
  3. 按标准结构逐模块生成
  4. 重点细化交互说明和字段定义

**用户输入**："我想做一个在线教育平台，老师可以上传课程，学生可以购买学习"

**智能体处理流程**：
1. 询问并确认：目标用户、核心功能、付费模式
2. 拆解模块：用户管理、课程管理、订单管理、学习记录
3. 生成"课程管理"模块的 PRD（5 部分结构）
4. 细化交互说明：
   - 查询条件：课程名称（模糊搜索）、分类（下拉）、状态（下拉）
   - 列表字段：课程名称、讲师、价格、状态、创建时间
   - 操作权限：新增、编辑、删除、上架/下架
   - 表单字段：课程标题（输入框，必填，限50字）、课程简介（文本域，选填，限500字）等

### 示例 2：从需求文档生成 PRD
- **功能说明**：用户上传需求文档，智能体提取信息并生成标准化 PRD
- **执行方式**：智能体主导（先读取文档，再生成 PRD）
- **关键指导要点**：
  1. 读取用户上传的需求文档
  2. 提取功能列表和关键细节
  3. 按标准 PRD 结构重新组织
  4. 补充缺失的字段定义和业务逻辑

**用户操作**：上传一份产品需求说明文档

**智能体处理流程**：
1. 读取文档内容，识别所有功能模块
2. 对每个模块，检查是否包含需求描述、前置条件、交互说明、后置条件、异常说明
3. 补充缺失的部分（如异常说明、字段限制）
4. 细化交互说明到字段级别
5. 生成符合标准的完整 PRD

### 示例 3：完善现有 PRD
- **功能说明**：用户已有初步 PRD，智能体帮助完善细节和规范格式
- **执行方式**：智能体主导（分析现有内容，补充缺失部分）
- **关键指导要点**：
  1. 读取现有 PRD
  2. 对比标准结构，识别缺失部分
  3. 补充字段定义、验证规则、异常处理
  4. 统一格式和术语

**用户输入**："这是我的 PRD，帮我检查和完善一下"（附上 PRD 文档）

**智能体处理流程**：
1. 读取 PRD，检查每个功能模块的 5 部分结构
2. 识别交互说明中不够细化的部分（如缺少字段限制）
3. 补充缺失的异常说明和后置条件
4. 统一字段描述格式
5. 输出完善后的 PRD
